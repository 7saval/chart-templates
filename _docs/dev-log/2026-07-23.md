### 2. 구현 코드 검토 — 가이드 스펙 대조 + 정적 검증

`IMPLEMENTATION_GUIDE.md` Phase 5b-2에 이미 검토를 마친 코드 스펙이 있어서, 실제 구현물을 한 줄씩 대조했다.

- `RankedList.stories.tsx`만 가이드 예시보다 확장됨: `TopWriters` 항목을 3개 → 5개로 늘려 `maxItems` 기본값(5) 경계에서 잘림 없이 그대로 나오는지 검증 가능하게 했고, `Empty` 스토리에 `emptyMessage: "No data available"`을 명시해 기본값이 아닌 커스텀 문구도 스토리로 확인 가능하게 했다. 둘 다 가이드가 요구한 "완료 체크" 항목(잘림 없음 확인)을 더 잘 커버하는 방향의 개선.
- Badge 배경색 인라인 스타일(`style={{ backgroundColor: STATUS_COLORS[item.status] }}` + `className="text-white"`) 패턴이 임의 선택이 아닌지 확인하려고 `AlertEventTable.tsx`를 대조했는데, 동일한 패턴(`SERVERITY_COLOR` 배경 + `text-white`)을 이미 쓰고 있어 기존 컨벤션과 일치함을 확인.
- `npx tsc -b` 실행 결과 `RankedList` 관련 에러는 0건. (`TopologyDiagram.tsx`의 d3 타입 에러 2건, `home.mock.ts`의 `severity`/`serverity` 오탈자 2건은 이번 작업과 무관한 기존 이슈로, `git status` 기준으로도 `RankedList` 폴더만 미추적 상태였다.)
- `npx eslint src/components/misc/RankedList/` 결과 `storybook/no-renderer-packages`(⁠`@storybook/react` 직접 import 금지) 규칙 위반 1건이 나왔는데, 같은 규칙 위반이 `MiniStatCell.stories.tsx`를 포함한 기존 스토리 파일 다수(`SparklineChart`, `PipelineFlowDiagram`, `KpiCardCompound`, `ProgressKpiCard`, `DashboardShell` 등)에도 동일하게 존재해 **레포 전역의 기존 lint 노이즈**로 확인됨 — `RankedList`가 새로 유발한 문제가 아님.
- 테스트 파일(`*.test.*`/`*.spec.*`)이 레포 전체에 하나도 없어, 이 프로젝트는 Storybook 스토리로 시각 검증을 대체하는 컨벤션임을 재확인. `RankedList`도 이 컨벤션을 그대로 따름.

**결론**: 별도 수정 없이 그대로 사용 가능. 정렬 로직을 컴포넌트가 갖지 않는 것(호출자가 이미 정렬된 배열을 넘긴다는 전제)과 `key={item.label}`을 키로 쓰는 것(라벨 중복 시 충돌 가능)은 가이드 설계 단계에서 이미 의도적으로 정한 부분이라 이번 구현의 문제로 잡지 않았다.

## 🐛 발생한 문제 & 해결

- 없음(RankedList 구현/검토 관련). 구현이 전날 확정해둔 가이드 스펙을 그대로 따라서 막힌 지점이 없었고, `tsc`/`eslint`에서 나온 이슈도 전부 이 작업과 무관한 기존 상태였다.

### 문제 2: `App.tsx`가 미완성 상태로 남아 있었음
- **원인**: 6-1(라우팅)까지는 구현됐지만 6-3(Home 조립) 작업 중간에 멈추면서 `Home`/`Kafka`/`Spark`/`PPSMinIO`/`useAutoRefresh`를 JSX/훅 호출에서만 참조하고 import를 추가하지 못한 상태로 남아 있었다. `Kafka`/`PpsMinIO` 페이지 자체가 없었으니 애초에 import할 대상도 없었다.
- **해결**: 4개 페이지와 `useAutoRefresh` 훅을 모두 구현한 뒤 `App.tsx`에 import 4개를 추가하고, JSX의 `<PPSMinIO />`를 실제 파일명(`PpsMinIO`)에 맞춰 통일했다.

### 문제 3: PPS/MinIO HPA 테이블의 스파크라인이 안 보임
- **원인**: `SparklineChart`(echarts, `width: '100%'`)를 `StatusDataTable`의 `render` 콜백에서 아무 래퍼 없이 직접 반환했다. 테이블 셀(`<td>`)은 명시적 너비가 없어 auto 폭으로 잡히는데, echarts는 마운트 시점 부모의 실제 렌더링 너비를 읽어 그리기 때문에 사실상 0폭으로 렌더링되어 빈 칸처럼 보였다.
- **해결**: `<div style={{ width: 80 }}>`로 감싸 고정 너비를 줬다. `KpiCard.tsx`가 `<div className="w-24 shrink-0">`로 감싸는 것과 동일한 이유.
- **참고**: 정적 타입체크/린트로는 잡히지 않는 런타임 레이아웃 버그라, `claude-in-chrome`으로 실제 브라우저 렌더링을 스크롤하며 육안 검증하지 않았다면 놓쳤을 것.

## 💡 배운 점 / 메모

- 전날처럼 구현 전에 "왜 이 props를 골랐는지"까지 문서화해두면, 구현 이후 검토 단계에서도 "스펙과 다른가"만 대조하면 되어 리뷰가 훨씬 빨라진다 — 설계 문서가 구현 가이드이자 리뷰 체크리스트 역할을 동시에 함.
- `tsc -b`/`eslint`를 컴포넌트 단위로 돌릴 때는 레포 전역에 걸친 기존 에러(다른 컴포넌트, mock 데이터)까지 같이 나오므로, 새로 작성한 파일 경로만 걸러서 봐야 "내가 만든 문제"와 "원래 있던 문제"를 헷갈리지 않는다.
- 규모가 큰 조립 작업(이번처럼 48개 섹션)을 시작하기 전에 "전 섹션 다 채울지 대표만 할지" 같은 스코프 질문을 먼저 던져두면, 중간에 범위를 다시 협의할 필요 없이 한 방향으로 쭉 진행할 수 있다.
- 타입체크/린트가 전부 통과해도 런타임 레이아웃 문제(테이블 셀 안 echarts 스파크라인이 폭을 못 잡는 것)는 안 잡힌다 — 차트/시각화 컴포넌트를 새로운 컨테이너(특히 `<td>`처럼 auto 폭인 곳)에 꽂을 때는 반드시 브라우저에서 실제로 렌더링해서 눈으로 확인해야 한다.
- 가이드 문서(`IMPLEMENTATION_GUIDE.md`)에 있는 예시 코드도 100% 정답은 아니다 — Home 2-5 섹션처럼 주석 라벨과 실제 내용이 어긋나 있는 경우가 있어, ch2~ch5 원본 스펙 문서와 대조하는 습관이 필요하다.
