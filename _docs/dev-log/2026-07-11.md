## 🐛 발생한 문제 & 해결

### 문제 1: `Badge` variant 타입 에러

- **원인**: `lucide-react`와 `@/components/ui/badge`에 동명의 `Badge`가 존재해, import 경로를 잘못 지정하면서 아이콘 컴포넌트가 사용됨.
- **해결**: import 경로를 shadcn/ui의 `Badge`로 수정.
- **참고**: 동명 컴포넌트가 여러 라이브러리에 있을 때 자동완성이 잘못된 경로를 잡아줄 수 있으므로 import 시 주의 필요.

### 문제 2: ALL 탭을 눌러도 전체 목록이 안 나옴

- **원인**: `TabsTrigger value="All"`와 필터 비교식 `filter === "ALL"`의 대소문자 불일치로 조건이 항상 거짓이 되어 `serverity === "All"`로 필터링됨 (매칭되는 값 없음).
- **해결**: `TabsTrigger`의 `value`를 `"ALL"`로 통일.
- **참고**: 문자열 리터럴 상수는 오탈자/대소문자 불일치에 취약하므로, 필요 시 enum이나 상수로 분리하는 것도 고려할 만함.

## 💡 배운 점 / 메모

- `NonNullable<T>`는 런타임 함수가 아니라 TypeScript 유틸리티 타입으로, `T extends null | undefined ? never : T`로 정의되어 있음. optional 필드를 `Record`의 키 타입으로 쓸 때 `undefined`를 제거하는 용도로 유용.
- 동일한 이름의 컴포넌트가 여러 패키지(`lucide-react` vs `@/components/ui/*`)에 존재할 수 있으므로, import 자동완성 결과를 한 번 더 확인하는 습관이 필요.
- 문자열 리터럴 값을 여러 곳(JSX prop, 비교 조건)에서 반복 사용할 때는 상수로 추출하면 이런 대소문자 불일치 버그를 예방할 수 있음.

### `@tanstack/react-table`을 언제 도입해야 하는가

`StatusDataTable`(가이드 4-1)을 뜯어보며 `@tanstack/react-table`이 실제로 대체하는 게 뭔지 정리했다.

- **핵심 아이디어 — "헤드리스(headless) 테이블"**: react-table은 UI를 렌더링하지 않는다. 정렬·필터·페이지네이션 등의 **상태와 로직**만 관리하고, 실제 `<table>`, `<th>`, `<td>` 마크업은 전부 우리가 shadcn `Table` 컴포넌트로 그린다. 그래서 스타일 시스템(shadcn/Tailwind)과 충돌 없이 로직만 가져다 쓸 수 있다.

- **`columnHelper.accessor(fn, config)`**: 컬럼이 "데이터의 어느 값을 보여줄지"와 "어떻게 렌더링할지"를 선언적으로 분리해서 정의하게 해준다.
  ```ts
  columnHelper.accessor((row) => row[col.key], {
    id: String(col.key),
    header: col.header,
    enableSorting: col.sortable ?? false,
    cell: (info) => ... // 커스텀 렌더링
  })
  ```
  `row[col.key]`처럼 접근 로직을 함수로 넘기기 때문에, `TableColumn<T>[]`라는 런타임 설정 배열을 그대로 react-table 컬럼 정의로 변환할 수 있다. 이게 `StatusDataTable`이 제네릭 `<T>`를 받아 어떤 데이터 모양에도 재사용 가능한 이유.

- **`getCoreRowModel()` / `getSortedRowModel()`**: react-table은 "row model"을 파이프라인처럼 조립하는 구조다. `getCoreRowModel()`은 원본 데이터를 행(row) 객체로 변환하는 기본 단계이고, `getSortedRowModel()`을 추가로 꽂으면 `sorting` state에 따라 정렬된 행 목록을 다시 계산해준다. 정렬 비교 함수(문자열/숫자 비교, asc/desc 토글)를 직접 짤 필요가 없다.

- **`enableSorting` / `getCanSort()` / `getToggleSortingHandler()`**: 컬럼 정의에서 `enableSorting: col.sortable ?? false`로 컬럼별 정렬 가능 여부를 켜고 끄면, 헤더 렌더링 쪽에서 `header.column.getCanSort()`로 "이 컬럼이 정렬 가능한가"를 물어보고 `header.column.getToggleSortingHandler()`를 `onClick`에 꽂기만 하면 클릭 정렬이 완성된다. 정렬 상태 토글(안 함 → asc → desc)과 아이콘 표시(`getIsSorted()`)까지 훅이 다 관리해준다.

- **왜 `AlertEventTable`엔 필요 없었나**: 컬럼이 5개로 고정, 정렬 요구사항 없음, 필터도 심각도 탭 하나뿐이라 `useState` + `Array.filter`로 끝난다. react-table의 진짜 값어치는 "컬럼 구성이 소비자마다 달라지는 제네릭 컴포넌트"에서 나오는데, `AlertEventTable`은 그 조건에 해당하지 않는다. 즉 **"정렬/필터가 필요하다"가 아니라 "컬럼 정의를 데이터와 분리해서 여러 화면에 재사용해야 한다"가 react-table 도입의 실제 기준**이라는 걸 이번에 명확히 정리했다.
