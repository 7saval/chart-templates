### 1. `TopologyDiagram` 구현 완료

`_docs/dev-log/2026-07-18.md`에서 "폴더만 있고 실제 컴포넌트 코드가 아예 없다"고 확인됐던 3개 컴포넌트(`MiniStatCell`, `RankedList`, `TopologyDiagram`) 중 `TopologyDiagram`을 직접 구현해서 완성했다. Ch3 Kafka 브로커 토폴로지(3-4)/Ch5 MinIO 토폴로지처럼 **방향성 없는 mesh network**를 그리는 컴포넌트로, Phase 5-0 설계 검토에서 결정한 대로 React Flow가 아니라 `d3-force`를 그대로 사용했다.

- `TopologyDiagram.types.ts`: `TopologyNode`(id/label/status/badges/isController), `TopologyEdge`(source/target), `TopologyDiagramProps`(nodes/edges/width/height) 정의.
- `TopologyDiagram.tsx`: `useEffect` 안에서 `d3.forceSimulation` + `forceLink`(거리 120) + `forceManyBody`(척력 -300) + `forceCenter`로 물리 시뮬레이션 구성. 노드는 원(circle) + 라벨(text)로 그리고, `isController`인 노드는 테두리를 강조. `d3.drag`로 노드를 직접 끌어서 위치를 고정(`fx`/`fy`)할 수 있게 했고, `simulation.on("tick", ...)`에서 매 프레임 위치를 갱신.
- 가이드 원본 코드는 링크/노드 콜백에서 `(d: any)`를 썼는데, 노드 id 콜백은 `(d: SimNode)`로 구체적인 타입을 지정해서 가이드보다 타입 안정성을 살짝 높였다. (`source`/`target`은 `forceLink`가 내부적으로 string id를 노드 객체로 치환하는 특성 때문에 tick 핸들러 쪽은 `any`를 그대로 둠 — d3-force 타입 정의의 알려진 한계.)
- `TopologyDiagram.stories.tsx`: Kafka 5-broker(`KafkaBrokers`), MinIO 8-node(`MinioNodes`) 목 데이터로 스토리 2개 작성.
- 참고 파일: `src/components/topology/TopologyDiagram/TopologyDiagram.tsx`, `TopologyDiagram.types.ts`, `TopologyDiagram.stories.tsx`

## 🐛 발생한 문제 & 해결

### 문제 1: `PipelineFlowDiagram` 안에서 스파크라인이 안 보임

- **원인**: 위 완료한 작업 2 참고 — 스토리 목 데이터에 `sparklineData` 필드 누락.
- **해결**: `PipelineFlowDiagram.stories.tsx`의 `Default`/`WithWarning` 노드 data에 트렌드 배열 추가.

### 문제 2: `TopologyDiagram.stories.tsx`에서 TS1355 (`const` assertion) 에러

```
A 'const' assertions can only be applied to references to enum members,
or string, number, boolean, array, or object literals.ts(1355)
```

- **원인**: `as const`는 문자열/숫자/불리언/배열/객체 리터럴에만 붙일 수 있는데, `(i === 6 ? "warning" : "normal") as const`처럼 **삼항식 전체**에 붙였음. 삼항식 자체는 리터럴이 아니라서 에러가 남.
- **참고**: 이 오타는 `IMPLEMENTATION_GUIDE.md` 원문(5-3 스토리 예시)에도 그대로 있던 것이라, 가이드를 충실히 베낀 결과였음 — 가이드 코드라고 무조건 신뢰하지 말고 컴파일 에러가 나면 실제 타입 규칙 기준으로 고쳐야 한다는 걸 확인.
- **해결**: 삼항식의 각 분기(`"warning"`, `"normal"`)에 개별적으로 `as const`를 적용하도록 수정.

## 💡 배운 점 / 메모

- `d3-force`처럼 시뮬레이션 도중 데이터 형태가 바뀌는 라이브러리(`forceLink`가 `source`/`target`을 string id에서 노드 객체로 치환)는 콜백 인자에 구체 타입을 끝까지 못 밀어붙이는 지점이 생긴다 — 가능한 곳(`id` 콜백)만 타입을 좁히고, 라이브러리 내부 동작 때문에 어쩔 수 없는 곳(`tick` 핸들러의 `source.x`/`target.x`)은 `any`로 남기는 절충이 실용적이다.
- 가이드 문서의 코드 블록 중 주석으로 "발췌"라고 표시된 것은 파일 전체가 아니다 — 그대로 복붙하면 import/선언부가 빠진 채로 남을 수 있으니, 붙여넣기 전에 "이게 파일 전체인지 일부인지"를 먼저 확인하는 습관이 필요.
- `as const`는 표현식 전체가 아니라 리터럴 각각에 적용해야 한다. 삼항식·함수 호출 결과처럼 "리터럴이 아닌 표현식"에 걸면 TS1355가 난다.
- 가이드/문서에 있는 코드라도 오타나 버그가 있을 수 있다. 컴파일 에러가 나면 "가이드를 따라 했으니 맞겠지"가 아니라 실제 타입 규칙을 기준으로 판단해야 한다.
