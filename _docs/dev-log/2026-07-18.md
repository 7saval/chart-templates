## ✅ 완료한 작업

### 1. 2주차 — 프롬프트 설계 + Claude API 연동

StoryDoc AI가 실제로 Claude한테 "이 컴포넌트 문서 좀 써줘"라고 요청하는 부분을 만들었다.

- **few-shot 예시**: AI한테 무작정 "문서 써줘"라고 시키면 형식이 제각각으로 나온다. 그래서 이미 프로젝트에 있는 잘 만들어진 Story 파일 3개(`TrendLineChart`, `KpiCard`, `AlertEventTable`)를 "이렇게 써줘"라는 참고 예시로 프롬프트에 통째로 넣어줬다. 몇 개의 실제 예시를 보여주고 그대로 따라 하게 만드는 기법을 few-shot이라고 부른다.
- **구조화된 출력(structured output)**: Claude가 자유 텍스트로 대답하면 나중에 파싱하기가 지저분해진다. 그래서 "이 형식(JSON 스키마)에 딱 맞춰서 대답해"라고 강제하는 기능(`tool_choice`)을 썼다. 결과물은 항상 `{ mdx: "문서 내용", storyVariants: [...] }` 형태로 고정된다.
- **`claude-sonnet-5` 모델 연동**: API 키는 코드에 직접 박아넣지 않고 `.env` 파일에서 읽어온다(보안 + 나중에 다른 모델/사내 모델로 바꾸기 쉽게).
- 검증: few-shot에 없던 `SparklineChart`로 실제 호출해서 확인 → Props를 정확히 뽑아내고, 없는 prop을 지어내지 않는 것 확인.
- 참고 파일: `tools/storydoc-ai/src/prompt/`, `tools/storydoc-ai/src/client/`

### 2. 3주차 — 실제 파일 생성 + 안전장치

2주차까지는 AI가 만든 내용을 그냥 JSON 파일로만 저장했다. 3주차부터 진짜로 `.stories.tsx`/`.mdx` 파일을 만들도록 했다.

- **코드 자동 정렬(Prettier)**: AI가 만든 코드가 들쭉날쭉하지 않게 자동으로 예쁘게 정렬.
- **덮어쓰기 방지**: 이게 제일 중요한 안전장치다. 이미 사람이 만들어둔 `.stories.tsx`가 있으면 **그 파일은 절대 건드리지 않는다.** 대신 "이렇게 바꾸면 어때?"라는 제안을 `.stories.ai.tsx.diff`라는 별도 파일로만 만든다. 파일이 아예 없던 컴포넌트는 새로 만들어도 된다(덮어쓸 게 없으니까).
- 참고 파일: `tools/storydoc-ai/src/render/`, `tools/storydoc-ai/src/postprocess/`

## 🐛 발생한 문제 & 해결 (내가 물어본 질문 순서대로)

### 문제 1: "이런 에러는 왜 나는거야?" — `@storybook/blocks` import 실패

```
Failed to resolve import "@storybook/blocks" from ".../SparklineChart.mdx"
```

- **원인**: AI가 만든 MDX 파일이 `@storybook/blocks`라는 패키지에서 `Meta`, `Canvas` 같은 걸 가져오려고 했는데, 이 프로젝트가 쓰는 Storybook 버전(10)에는 그 패키지가 없다. 실제로는 `@storybook/addon-docs/blocks`라는 다른 경로에 들어있다. AI가 학습해둔 옛날 Storybook 컨벤션을 그대로 재현해버린 것.
- **해결**: ① 이미 만들어진 파일의 import 줄을 직접 고침. ② 앞으로 AI가 같은 실수를 반복하지 않도록 프롬프트에 정확한 import 경로를 명시적으로 지시 추가.

### 문제 2: "표처럼 표시 안되어 있다" — Props 표가 그냥 텍스트로 보임

MDX 문서에 넣은 Props 표가 `| Name | Type | ... |` 형태 그대로 줄글로 나오고, 진짜 표(칸이 나뉜 모양)로 안 보임.

- **원인**: 마크다운에서 `|`로 표를 그리는 문법은 사실 "표준"이 아니라 GitHub이 추가로 만든 확장 기능(GFM, GitHub Flavored Markdown)이다. 이걸 해석해주는 `remark-gfm`이라는 플러그인이 이 프로젝트 Storybook 설정에 아예 없었다. 그래서 AI가 만든 것도, 사람이 손으로 써도 어차피 똑같이 안 그려지는 상태였다 — AI만의 문제가 아니라 레포 설정 자체의 빈틈이었다.
- **해결**: 레포 전체 설정(`.storybook/main.ts`)에 `remark-gfm`을 설치하고 연결. 이건 storydoc-ai 도구만이 아니라 레포 전체에 영향을 주는 수정이라, 진행 전에 사용자에게 먼저 확인받고 진행했다.

### 문제 3: "Error fetching `/index.json`... 왜?" — Storybook 전체가 안 뜸

- **원인**: MDX 파일에서 `<Meta title="..." of={...} />`처럼 `title`과 `of`를 동시에 썼다. `of`를 쓰면 Storybook이 알아서 연결된 `.stories.tsx`의 title을 가져오는데, 여기에 `title`을 또 직접 쓰면 같은 이름의 항목이 두 개 생기는 셈이 돼서 전체 스토리 목록(index)을 만드는 과정 자체가 깨져버렸다.
- **해결**: `title`을 빼고 `of`만 남김. 프롬프트에도 같은 실수를 반복하지 않도록 지시 추가.

### 문제 4: "이걸 만든 근거는 무엇인가?" — 존재하지 않는 `label` 필드로 타입 에러

```ts
Object literal may only specify known properties, and 'label' does not exist in type 'PipelineFlowNodeType'.
```

- **원인**: `PipelineFlowDiagram`의 `nodes` prop은 사실 `@xyflow/react`라는 외부 라이브러리의 복잡한 타입(`Node<PipelineFlowNodeProps, "pipelineNode">`)을 쓰고 있다. 그런데 storydoc-ai가 컴포넌트 코드를 분석할 때 쓰는 도구(`react-docgen-typescript`)는 이런 "타입 안에 다른 타입이 들어있는 구조"까지는 못 들여다본다. 그래서 AI는 `nodes` 안에 정확히 어떤 필드가 들어가야 하는지 전혀 모른 채로, "플로우 다이어그램 노드라면 이럴 것 같다"고 그럴듯하게 지어냈다(`label`, `type: 'producer'` 등 — 실제로는 존재하지 않는 필드들). 이렇게 정보가 없는 상태에서 그럴듯하게 지어내는 걸 hallucination(환각)이라고 부른다.
- **해결**: 실제 타입 정의 파일을 직접 읽어서 진짜 구조(`data: { name, metrics, status }` 형태로 감싸져 있음)를 확인하고 사람이 직접 고쳤다. 그리고 이 문제가 이 컴포넌트만의 일이 아니라 19개 중 11개 컴포넌트에서 똑같이 발생할 수 있다는 걸 확인해서, 계획 문서에 "위험군" 목록으로 남겨뒀다.

### 문제 5: "밑에 잘려서 보이는 버그" — 노드 3개가 계단식으로 잘림

- **원인 파악 과정**: 헤드리스 브라우저(Playwright, 화면 없이 동작하는 브라우저)를 그 자리에서 설치해서 직접 화면을 캡처해봤다. 노드 3개가 나란히 한 줄로 있어야 하는데 계단처럼 아래로 밀리면서 화면 밖으로 잘려 있었다. 각 노드의 실제 렌더링 좌표를 찍어보니, 위쪽 노드의 "아래 끝"과 바로 아래 노드의 "위 끝"이 정확히 맞닿아 있었다 — 이건 "지정한 위치에 배치된 것"이 아니라 그냥 위에서 아래로 순서대로 쌓인 것(HTML의 기본 동작)이라는 뜻이었다.
- **진짜 원인**: `PipelineFlowDiagram`이 쓰는 `@xyflow/react` 라이브러리는 노드를 원하는 위치에 배치하려면 자기 전용 CSS 파일이 반드시 로드되어 있어야 한다. 실제 앱(`src/main.tsx`)에는 이 CSS가 잘 들어있었는데, **Storybook은 앱과 별도의 진입점(`.storybook/preview.tsx`)을 쓰기 때문에 그 CSS가 로드되지 않고 있었다.** 이 컴포넌트는 이번에 처음으로 Storybook에 등록됐기 때문에 지금까지 아무도 발견하지 못했던 문제.
- **해결**: `.storybook/preview.tsx`에 CSS import 한 줄(`import '@xyflow/react/dist/style.css'`)을 추가. 다시 캡처해보니 세 노드가 정확히 한 줄로 정렬됨.

### (참고) "loading props를 안 만드는 건 어떤 판단?" — 버그는 아니고 의도된 규칙

- 프롬프트에 "실제로 존재하는 prop만 가지고 variant를 만들어라, 없는 걸 지어내지 마라"는 규칙이 있다. `PipelineFlowDiagram`은 최상위 prop에 `isLoading`이 없어서 규칙대로 `Loading` variant를 안 만든 게 맞다. 다만 실제로는 노드 하나하나의 데이터 안에 `isLoading`이 숨어 있는데(문제 4와 같은 이유로 도구가 못 봄), 요청에 따라 `Loading` variant를 수동으로 추가해서 스켈레톤 로딩 화면이 정상 표시되는 것까지 확인함.

### 문제 6: "컴포넌트에 비해 Canvas 영역이 너무 크게 느껴짐" — 작은 컴포넌트도 화면 전체 크기로 나옴

- **원인**: `.storybook/preview.tsx`의 전역 decorator(모든 스토리를 감싸는 공통 wrapper)에 Tailwind의 `min-h-screen`(최소 높이 = 화면 전체 높이) 클래스가 걸려 있었다. 이게 **모든 스토리에 예외 없이** 적용되다 보니, `SparklineChart`처럼 실제로는 32px밖에 안 되는 작은 컴포넌트도 항상 화면 전체 높이만큼의 빈 배경 박스 안에 렌더링되고 있었다.
- **확인 과정**: 다른 컴포넌트 중에 `layout: "fullscreen"`처럼 크기를 직접 지정한 게 있는지 찾아보니 `DashboardShell` 하나뿐이었다. `DashboardShell`은 컴포넌트 자체에 이미 `h-screen`이 박혀 있고 스토리에도 `fullscreen` 설정이 따로 있어서, 전역 `min-h-screen`을 빼도 이 컴포넌트는 영향받지 않는다는 걸 확인하고 안전하게 제거했다.
- **해결**: `min-h-screen` 삭제. Playwright로 다시 캡처해서 실제 렌더링 높이가 112px(컴포넌트 크기에 맞는 값)로 줄어든 것 확인.
- **삽질 포인트**: 처음에 800px 높이 브라우저 창으로 스크린샷을 찍었더니 여전히 크게 나와서 "안 고쳐졌나?" 싶었는데, 이건 스크린샷 도구(`fullPage: true`)가 실제 콘텐츠가 브라우저 창보다 작으면 빈 공간까지 창 크기만큼 채워서 찍는 특성 때문이었다. 브라우저 창을 작게(150px) 줄여서 다시 찍어보니 제대로 줄어든 게 보였다. → **화면 캡처 도구 자체의 동작 방식 때문에 "고친 게 반영이 안 됐다"고 착각할 수 있으니, 실제 DOM 요소 크기(`getBoundingClientRect()`)를 같이 확인하는 습관이 필요.**

### (참고) "diff는 반영하고 지워도 되나? 다시 돌리면 diff도 안 덮어써지나?" — 코드로 정확히 확인한 질문

- `.stories.ai.tsx.diff` / `.ai.mdx.diff` 파일은 사람이 읽으라고 만드는 일회성 참고 자료일 뿐, 다른 코드가 다시 읽어서 참조하는 파일이 아니다. 내용을 실제 파일에 반영했으면 **지워도 안전**.
- `writeOrDiff.ts` 코드를 다시 확인해보니, "덮어쓰지 않기" 보호 로직은 **원본 파일(`.mdx`/`.stories.tsx`)에만** 걸려 있고, `.diff` 파일 자체는 존재 여부 확인 없이 **매번 무조건 새로 씀**. 즉 이전 diff를 안 지워도 다음 실행에서 그냥 덮어써진다 — "diff 파일도 보호되는 거 아니냐"는 예상과는 다르게, diff는 애초에 보호 대상이 아니라 매번 새로 계산되는 결과물이라는 게 정확한 답.

## 💡 배운 점 / 메모

- 마크다운 표(`| a | b |`)는 사실 표준 마크다운이 아니라 GFM이라는 확장 기능이고, 프로젝트마다 지원 여부가 다를 수 있다.
- AI가 코드를 생성할 때, 그 프로젝트의 실제 최신 설정을 미리 안 보여주면 "일반적으로 많이 쓰는 옛날 방식"을 그대로 재현해버린다 — few-shot에 넣을 예시를 고를 때는 "실제로 동작이 검증된 예시"인지가 중요하다.
- 타입이 다른 타입을 감싸고 있는 구조(제네릭, `Node<T>` 같은 것)는 코드 분석 도구가 속을 못 들여다보는 경우가 많다. 이럴 땐 AI가 안전하게 지어내지 못하도록 아예 "이 정보는 없다"는 걸 명확히 하거나, 위험군으로 표시해서 사람이 한 번 더 확인하게 만드는 게 낫다.
- 화면이 이상하게 보일 때 "왜 이렇게 배치됐지?" 싶으면, 브라우저에서 실제 렌더링된 요소의 좌표(`getBoundingClientRect()`)를 찍어보면 원인을 훨씬 빨리 좁힐 수 있다.
- 앱의 진입점(`main.tsx`)과 Storybook의 진입점(`.storybook/preview.tsx`)은 서로 다른 파일이라, 한쪽에만 import해둔 CSS/설정이 다른 쪽에 자동으로 넘어가지 않는다 — 새로운 외부 라이브러리 기반 컴포넌트를 Storybook에 처음 등록할 때는 이 부분을 꼭 확인해야 한다.
- "모든 화면에 공통으로 적용되는 설정"(전역 decorator, 전역 CSS 등)은 특정 상황(예: 전체 화면을 차지하는 레이아웃 컴포넌트)을 염두에 두고 넣은 거라도, 정작 그 설정은 관련 없는 다른 모든 컴포넌트에도 똑같이 적용된다는 걸 잊기 쉽다. 전역 설정을 만질 땐 "이걸 진짜로 필요로 하는 컴포넌트가 몇 개나 되는지" 먼저 확인하는 게 좋다.
- 브라우저 스크린샷 도구의 `fullPage` 옵션은 "실제 콘텐츠 크기"가 아니라 "브라우저 창 크기 이상"으로 찍힌다 — 요소가 작아 보이는지 확인할 땐 스크린샷보다 `getBoundingClientRect()` 같은 실제 측정값을 우선 신뢰하는 게 낫다.
- `.diff` 같은 "사람이 검토용으로만 보는 산출물"은 굳이 보호 로직을 걸 필요가 없다 — 매번 새로 계산해서 던져주고, 검토 끝나면 지워도 되는 소모품으로 설계하는 게 오히려 더 단순하고 안전하다.
