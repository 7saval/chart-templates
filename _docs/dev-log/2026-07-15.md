### 2. `tools/storydoc-ai/` 스캐폴딩

프론트엔드 빌드(Vite)와 완전히 분리된 독립 Node CLI 패키지로 만들었다. 계획 문서 3장에서 "pnpm 모노레포"가 아니라 "레포 루트에 자체 package.json을 가진 독립 폴더"로 시작하기로 한 결정을 그대로 따름.

- `tools/storydoc-ai/package.json` — `type: module`. 런타임 의존성은 `commander`(CLI 프레임워크), `react-docgen-typescript`(AST 파싱). devDependency로 `typescript ~6.0.2`(루트와 버전 통일), `tsx`(ESM+TS를 별도 로더 설정 없이 바로 실행 — `ts-node`는 ESM 모드에서 추가 플래그가 필요해서 배제), `@types/node`. `dev`(tsx로 즉시 실행)/`build`(tsc 컴파일)/`start`(컴파일 결과 실행)/`typecheck` 스크립트 구성.
- `tools/storydoc-ai/tsconfig.json` — 프론트엔드용 `tsconfig.app.json`(module: bundler, noEmit)과는 별개의, 실제로 Node에서 실행되는 CLI에 맞는 독립 설정. `module`/`moduleResolution: NodeNext`로 지정하고 루트 `tsconfig.json`의 `references` 배열에는 추가하지 않아서 루트 빌드(`tsc -b && vite build`)에 영향이 없도록 격리.
- `tools/storydoc-ai/.gitignore` — `node_modules`, `dist`, `.output/`. 특히 `.output/`은 이번 주 새로 생기는 JSON 덤프 산출물 디렉토리라 루트 `.gitignore`에는 없는 항목이라 별도로 추가.

### 3. AST 파서 래퍼 — `src/parser/index.ts`

`react-docgen-typescript`를 직접 CLI 곳곳에서 호출하지 않고, `createParser` / `parseComponent` 두 함수로 감싼 이유는 파서 설정(tsconfig 경로, 옵션)을 한 곳에서만 관리하고 2주차 이후 여러 컴포넌트를 순회할 때 파서 인스턴스를 재사용하기 위함.

- `createParser({ tsconfigPath })` — `withCustomConfig`를 호출하는데, **새 tsconfig를 만들지 않고 레포 루트의 `tsconfig.app.json`을 그대로 가리키게** 했다. 이 파일에 이미 `@/*` 경로 별칭, `jsx: react-jsx` 등 실제 컴포넌트들이 의존하는 설정이 다 들어있어서, 파서 전용 설정을 새로 만들면 "실제 빌드에 쓰이는 설정"과 "파서가 참조하는 설정"이 따로 놀 위험(별칭 drift)이 있었기 때문.
- 옵션: `shouldExtractValuesFromUnion: true`(유니온 타입의 실제 값 추출), `shouldExtractLiteralValuesFromEnum: true`(TS `enum` 대응, 이번 주 두 컴포넌트엔 해당 사항 없었지만 이후 21개 컴포넌트 중 있을 수 있어 forward-compatibility로 포함), `shouldRemoveUndefinedFromOptional: true`, `propFilter`로 `node_modules`에서만 유래한 prop(예: 순수 DOM/React 상속 속성) 제외해서 덤프가 컴포넌트 자체 prop만 담도록 함.
- `parseComponent(parser, componentFile)` — `parser.parse(file)`은 배열을 반환하는데, 파일 하나에 컴포넌트 하나만 있다는 전제로 첫 번째 결과만 꺼내고, 없으면 명시적으로 에러를 던지게 함(파싱 실패를 조용히 넘기지 않도록).

### 4. CLI 진입점 — `src/cli.ts`, `src/commands/parse.ts`, `src/paths.ts`

- `src/paths.ts` — `TOOL_DIR`(패키지 루트), `REPO_ROOT`(chart-templates 루트)를 `import.meta.url` 기준으로 계산. dev 모드(`tsx src/cli.ts`)와 빌드 모드(`node dist/cli.js`) 양쪽 다 파일이 패키지 루트에서 정확히 한 단계 아래에 위치하기 때문에 동일한 `../..` 두 번으로 두 모드 모두에서 올바른 경로가 나옴.
- `src/commands/parse.ts` — `storydoc-ai parse <componentPaths...>` 커맨드. 컴포넌트 경로를 여러 개(variadic) 받아 한 번의 실행으로 여러 파일을 처리하도록 설계 — 이번 주 목표가 "2개 컴포넌트 검증"이기도 하고, 4주차에 21개를 한 번에 돌릴 걸 미리 염두에 둠. 경로는 `REPO_ROOT` 기준 상대경로로 받아서 어느 디렉토리에서 실행하든 동일하게 동작. 파일 단위로 `try/catch`를 걸어서 하나가 실패해도 나머지는 계속 처리(4주차 일괄 실행 대비). 결과는 `tools/storydoc-ai/.output/<ComponentName>.json`에 prop명/타입/필수여부/기본값/description을 담아 저장.
- `src/cli.ts` — `commander`로 `parse` 커맨드를 등록하는 얇은 진입점.

## 🐛 발생한 문제 & 해결

### 문제 1: optional prop 타입이 전부 `"enum"`으로 뭉개짐

- **원인**: 파서 옵션에 `shouldExtractValuesFromUnion: true`를 켠 상태에서, TypeScript는 `boolean`을 내부적으로 `true | false` 유니온으로, optional prop(`T?`)은 `T | undefined` 유니온으로 표현한다. `react-docgen-typescript`가 이 유니온을 감지하면 `type.name`을 무조건 `"enum"`으로 바꿔버리고 실제 값은 `type.value`(원소 배열)와 `type.raw`(원본 타입 텍스트)에만 남긴다. 그 결과 `KpiCard.json`의 `isLoading`(`boolean`), `error`(`string`), `TrendLineChart.json`의 `height`/`isLoading`/`error`/`theme`가 전부 `"enum"`으로 출력됐다.
- **디버깅 과정**: `tools/storydoc-ai/debug.mjs`(임시 스크립트)로 `TrendLineChart`의 `height`/`isLoading`/`theme` 세 prop의 `type` 객체 전체를 JSON으로 찍어봐서, `name`은 `"enum"`이지만 `raw` 필드에 각각 `"string | number"`, `"boolean"`, `"\"dark\" | \"light\""`라는 원본 타입 텍스트가 그대로 남아있는 걸 확인했다. 확인 후 `debug.mjs`는 삭제.
- **해결**: `src/commands/parse.ts`의 JSON 매핑에서 `prop.type?.name` 대신 `prop.type?.raw ?? prop.type?.name ?? null`을 쓰도록 수정. 재실행 결과 `isLoading: "boolean"`, `error: "string"`, `height: "string | number"`, `theme: "\"dark\" | \"light\""`로 정상 출력됨을 확인.
- **참고**: `height`가 계획 문서 예시(5장)에서는 `number`로 나오지만 실제 `ChartBaseProps`에서는 `string | number`로 정의되어 있었다 — 문서 예시와 실제 타입 정의가 다른 경우가 있다는 것도 이번에 확인.

## 💡 배운 점 / 메모

- `react-docgen-typescript`에서 `shouldExtractValuesFromUnion`을 켤 때는 `type.name`이 유니온 전반(리터럴 유니온뿐 아니라 `boolean`, optional 처리로 생기는 `T | undefined`까지)에 대해 `"enum"`으로 뭉개진다는 점을 미리 알고 있어야 한다. 사람이 읽을 타입 텍스트가 필요하면 `type.raw`를 쓰는 게 안전하다.
- Props가 `extends`로 상속되는 패턴(`ChartBaseProps` → `TrendLineChartProps`)은 `react-docgen-typescript`가 알아서 평탄화해서 최상위 prop 목록에 포함시켜준다. 반면 `KpiCardProps.data: KpiCardData`처럼 다른 인터페이스를 "값 타입"으로 참조하는 경우는 재귀적으로 펼쳐지지 않고 `type: "KpiCardData"`라는 타입명 문자열로만 남는다 — 이 차이를 2주차 프롬프트 설계 시 감안해야 함(중첩 필드까지 문서화하려면 별도 처리 필요).
- 계획 문서(설계 시점의 가정)와 실제 코드 상태가 어긋나는 지점(`.prettierrc` 없음, `height` 타입 등)은 구현 전에 먼저 훑어서 확정하고 시작하는 게, 구현 중간에 발견해서 되돌아가는 것보다 훨씬 적은 마찰로 진행됐다.
