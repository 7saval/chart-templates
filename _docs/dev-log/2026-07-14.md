## 🔧 주요 변경사항

```
PipelineFlowNode.types.ts, PipelineFlowDiagram.types.ts
  interface PipelineFlowNodeProps { ... }        →  type PipelineFlowNodeProps = { ... }
  interface PipelineFlowEdgeData { ... }          →  type PipelineFlowEdgeData = { ... }
```
`@xyflow/react`의 `Node<T>`/`Edge<T>` 제네릭 제약(`Record<string, unknown>`)을 만족시키기 위해 두 타입 모두 `interface`에서 `type`으로 변경.

## 🐛 발생한 문제 & 해결

### 문제 1: `TS2344` — "Type does not satisfy the constraint 'Record<string, unknown>'"

- **원인**: `Node<T>`/`Edge<T>` 제네릭은 `T extends Record<string, unknown>`을 요구하는데, TypeScript는 `interface`를 선언 병합(declaration merging)이 가능한 "열린" 타입으로 취급해서 인덱스 시그니처(`[key: string]: unknown`)를 명시하지 않는 한 이 제약을 만족한다고 인정하지 않음. 반면 `type` 별칭(object literal)은 구조적으로 "닫힌" 타입이라 동일한 필드 구성이어도 제약을 통과함.
- **해결**: `PipelineFlowNodeProps`, `PipelineFlowEdgeData` 두 타입 선언을 `interface` → `type`으로 변경. 다른 곳에서 `extends`로 상속받아 쓰는 코드가 없어서 부작용 없이 해결됨.
- **참고**: React Flow(xyflow) v12 공식 문서에도 노드/엣지 data 타입은 `type`으로 선언할 것을 권장.

---

## 💡 배운 점 / 메모

- TypeScript에서 `interface`와 `type`은 겉보기엔 같은 필드를 가져도, 제네릭 제약(`extends Record<string, unknown>`) 검사에서는 다르게 취급된다. `interface`는 선언 병합 가능성 때문에 "닫힌 타입"으로 간주되지 않음.
- 라이브러리(xyflow 등)에서 제네릭에 커스텀 데이터 타입을 넘길 때는 습관적으로 `interface` 대신 `type`을 쓰는 게 안전.
- `PipelineFlowConnector`(수동 SVG)와 `PipelineFlowDiagram`(React Flow)을 당분간 병행 두되, 스토리북 타이틀/설명에 "대체 예정"임을 명시해두면 나중에 헷갈리지 않음.
